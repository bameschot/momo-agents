# STORY-013: TocBuilder — Table of Contents Generator

**Index**: 13
**Attempts**: 1
**Design ref**: design/md-to-pdf-cli.new.md
**Depends on**: STORY-012

## Context
The PDF output includes a clickable Table of Contents sidebar containing H1–H3 headings. Each entry is an anchor link to the corresponding heading in the document. `wkhtmltopdf` honours internal `#slug` links when invoked with `--enable-internal-links`, so the ToC entries become true in-document jump links in the final PDF. This story implements `build_toc()` which takes the ordered `list[Heading]` from `ParseResult` and produces a nested `<nav>` HTML string.

## Acceptance Criteria
- [ ] `build_toc(headings: list[Heading]) -> str` is implemented.
- [ ] Returns an empty string (`""`) when `headings` is empty (no sidebar rendered when the document has no H1–H3 headings).
- [ ] For non-empty `headings`, returns a `<nav>` element containing a nested `<ul>` structure:
  - H1 headings are top-level `<li>` items.
  - H2 headings are nested one level inside the nearest preceding H1 (or at the top level if no H1 has appeared yet).
  - H3 headings are nested one level inside the nearest preceding H2 (or H1 if no H2 has appeared).
- [ ] Each `<li>` contains an `<a href="#<slug>">` link with the heading's plain text as the link label.
- [ ] Slug uniqueness: the slugs in `Heading.slug` values (generated by the block parser in STORY-011) are used directly — `build_toc()` does not re-generate slugs.
- [ ] The `<nav>` element has a class or id suitable for CSS targeting (e.g. `<nav id="toc">`).

## Implementation Hints
- Build the nested structure by tracking the "current depth" as you iterate through headings. A stack-based approach works well: push/pop nesting levels as heading levels increase/decrease.
- The nesting rules mirror the companion `md2html.py` ToC builder — refer to that implementation if already completed.
- Do not add `<details>`/`<summary>` wrappers for the PDF ToC — those are interactive HTML elements that `wkhtmltopdf` may not render usefully. A plain `<nav>` with a nested `<ul>` is sufficient.
- The CSS (STORY-014) will style this as a fixed sidebar; the ToC HTML itself just needs clean, correct nesting.

## Test Requirements
- Empty list → `build_toc([]) == ""`.
- Single H1 → `<nav>` with one `<li>` containing `<a href="#slug">Title</a>`.
- H1 + H2 + H3 → correct three-level nesting: H2 inside H1's `<ul>`, H3 inside H2's `<ul>`.
- H2 with no preceding H1 → H2 appears at the top level (not inside a non-existent H1 container).
- Multiple H2s under the same H1 → all appear as siblings in the same nested `<ul>`.
- Duplicate slugs (e.g. `foo` and `foo-2`) → both `href` values appear correctly in the output.
- H4 headings (which do not appear in `ParseResult.headings`) → not present in ToC output.
